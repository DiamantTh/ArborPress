TAGS vs. KATEGORIEN
───────────────────
  Tags        = flache Stichworte, frei vergeben, viele pro Post
                Beispiel: "python", "tutorial", "2026", "release-notes"
  Kategorien  = hierarchische Taxonomie, redaktionell gepflegt, 1–2 pro Post
                Beispiel: Technik → Python → Tutorial (Baum via parent_id)
  Beide Systeme existieren parallel und ergänzen sich.


PGP MULTI-SCHLÜSSEL
────────────────────
  Ein Nutzer kann mehrere OpenPGP-Schlüssel hinterlegen (user_pgp_keys).
  Jeder Schlüssel hat unabhängige Rollen:
    use_for_signing    → ausgehende Mails DIESES Nutzers werden damit signiert
    use_for_encryption → eingehende Mails AN diesen Nutzer werden damit verschlüsselt
  Primär-Signierschlüssel: is_primary_signing=TRUE; nur einer pro Nutzer.
  Fingerprint als eindeutiger Identifikator (unique per Nutzer).

TABLE: post_access_tokens   [Passwortschutz – mehrere Token pro Post]  [NEU]
===============================================================================
  id                    VARCHAR(36)   PK   UUID
  post_id               VARCHAR(36)   FK → posts.id   ON DELETE CASCADE
  label                 VARCHAR(128)       z. B. "Pressezugang", "Beta-Leser"
  token_hash            VARCHAR(256)       Argon2/bcrypt-Hash des Klartexts
  max_uses              INT           NULL  NULL = unlimitiert
  use_count             INT                Anzahl erfolgreicher Einlösungen
  expires_at            DATETIME      NULL  NULL = kein Ablauf
  created_at            DATETIME

  IDX: post_id  → ix_post_access_tokens_post_id

  Ablauf:
    1. Besucher ruft /p/<slug> auf → Passwort-Formular
    2. Eingabe → Backend prüft alle token_hash-Einträge des Posts
    3. Treffer + is_valid() → kurzlebiges Cookie → Inhalt sichtbar
    4. use_count wird erhöht; max_uses-Bedingung geprüft

  is_valid() = expires_at nicht überschritten UND
               (max_uses NULL ODER use_count < max_uses)

LESEZEIT-BERECHNUNG (reading_time_min)                                  [NEU]
===============================================================================

  Formel (berechnet beim Speichern, gespeichert als INT):
    1. Code-Blöcke (```...```) extrahieren und aus Prose entfernen
    2. Prose-Wörter zählen (split auf Whitespace)
    3. reading_time = prose_words / 200 + len(code_blocks) * 0.5
    4. round() + max(1, …) → Minimum 1 Minute

  Hintergrund:
    200 wpm = Durchschnitt Erwachsene (deutsche Prose)
    × 0.5 pro Code-Block = Code wird ca. halb so schnell überflogen
